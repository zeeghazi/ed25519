X25519 and Ed25519 are often confused because both are built on the same
elliptic curve, **Curve25519**. The difference is what they are _for_: **Ed25519
signs and verifies; X25519 establishes a shared secret.**

## The one-line difference

|             | Ed25519                             | X25519                                   |
| ----------- | ----------------------------------- | ---------------------------------------- |
| Job         | Digital **signatures** (EdDSA)      | **Key exchange** (Diffie-Hellman / ECDH) |
| Answers     | "Did this person really sign this?" | "Can we agree on a shared secret?"       |
| Curve       | Curve25519 (Edwards form)           | Curve25519 (Montgomery form)             |
| Typical use | SSH keys, JWT/EdDSA, code signing   | TLS 1.3 handshakes, encrypted messaging  |

## What Ed25519 does

Ed25519 is a **signature scheme**. You hold a private key and produce a
signature over a message; anyone with your public key can verify that you signed
it and that the message was not altered. It does not encrypt anything. This is
the algorithm behind [Ed25519 SSH keys](/ed25519-ssh-key/) and is what our
[in-browser signing tool](/ed25519-sign-message/) demonstrates. For a fuller
intro, see [what is Ed25519](/blog/what-is-ed25519/).

## What X25519 does

X25519 is a **key-agreement function**. Two parties each have a key pair and can
combine their own private key with the other's public key to compute the _same_
shared secret — without ever sending that secret over the wire. That shared
secret then keys a symmetric cipher. X25519 is the workhorse of TLS 1.3 and
modern messaging protocols like Signal.

## Same curve, different curve form

Both operate on Curve25519, but they use different representations of it:
Ed25519 uses the twisted **Edwards** form (efficient for signatures), while
X25519 uses the **Montgomery** form (efficient for Diffie-Hellman). The two are
mathematically related, and libraries such as libsodium can convert an Ed25519
key to its X25519 counterpart — but in practice you should generate and use a
**separate** key for each purpose rather than reusing one across both.

## Which one do you need?

- Proving identity or integrity (SSH login, signing a release, a JWT)? →
  **Ed25519**.
- Negotiating an encryption key for a session (HTTPS, a secure channel)? →
  **X25519**.

In many protocols you use both: X25519 sets up the encrypted channel, and
Ed25519 authenticates who you are talking to.
