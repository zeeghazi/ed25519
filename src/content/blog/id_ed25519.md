*Last updated: June 17, 2026*

If you have ever configured SSH (Secure Shell) access to a remote server, cloned a repository from GitHub or GitLab, or set up secure communications for an automated server environment, you have likely run into two files: `id_ed25519` and `id_ed25519.pub`. 

> **Featured Snippet: What is id_ed25519?**
> The file `id_ed25519` is the private key file of an Ed25519 SSH keypair, which must be kept secret on your client machine. The matching file `id_ed25519.pub` is the public key file, which can be shared publicly and uploaded to remote servers to enable passwordless authentication.


These files are located in your user directory's hidden `.ssh` folder and form the foundation of your digital identity on the network. But what exactly are these files, what is written inside them, and how do you secure them? 

In this comprehensive guide, we will break down the roles of the private and public key files, decode their internal formats, explain how to manage them using the SSH agent, look at setting secure folder permissions on both Linux and Windows, and troubleshoot common authentication errors.

## The Public and Private Key Pair Concept

At the core of SSH authentication is **asymmetric cryptography** (also known as public-key cryptography). Unlike symmetric cryptography, which uses a single shared password to both encrypt and decrypt data, asymmetric cryptography uses a matched pair of mathematically linked keys:

1. **The Private Key (`id_ed25519`):** This is your secret passkey. You must keep this file secure on your local computer and never share it with anyone.
2. **The Public Key (`id_ed25519.pub`):** This is your identity card. You can share this file freely with the world and install it on any server you wish to log into.

### Summary Comparison

| File Name | Key Type | Share It? | Local Permissions | Purpose |
| :--- | :--- | :--- | :--- | :--- |
| **`id_ed25519`** | Private Key | **Never** | Read/Write by owner only | Sign challenges to prove identity |
| **`id_ed25519.pub`** | Public Key | Yes, freely | Read/Write by all | Verify signatures sent by client |

### How Authentication Works (The Handshake)
When you attempt to connect to a remote server, the server does not ask for your private key. Instead, it performs a cryptographic challenge-response handshake:
1. The server checks its `authorized_keys` file to see if your public key (`id_ed25519.pub`) is registered.
2. The server generates a random challenge message and sends it to your client.
3. Your SSH client uses your local private key (`id_ed25519`) to sign the challenge, creating a digital signature.
4. Your client sends the signature back to the server.
5. The server uses your public key (`id_ed25519.pub`) to verify the signature. Because the signature could only have been created by the matching private key, the server trusts your identity and grants access.

## Where Do the Files Live?

By default, SSH client utilities write and look for keys in a hidden subdirectory named `.ssh` located in your user's home directory.

* **Linux, macOS, and WSL:**
  `/home/username/.ssh/` (often abbreviated as `~/.ssh/`)
* **Windows (Native OpenSSH):**
  `C:\Users\username\.ssh\` (often abbreviated as `%USERPROFILE%\.ssh\`)

If you run `ls -la ~/.ssh` (on Unix) or `dir /a %USERPROFILE%\.ssh` (on Windows), you will see both files listed alongside other SSH configuration files.

## How to Create the Key Files

To generate these files, use the OpenSSH utility called `ssh-keygen`. Open your terminal or PowerShell and run:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

* `-t ed25519`: Specifies the Ed25519 elliptic-curve algorithm.
* `-C "your_email@example.com"`: Appends an email comment to the end of the public key file, making it easy to identify.

During generation, you will see two prompts:
1. **Enter file in which to save the key:** Press Enter to accept the default location (`~/.ssh/id_ed25519`).
2. **Enter passphrase:** Enter a secure password to encrypt your private key file. While you can press Enter to leave it empty, setting a passphrase is highly recommended to protect your key if your physical device is lost or stolen.

For a full step-by-step walkthrough of generating keys, adding them to GitHub, and managing configurations, see our detailed [Ed25519 SSH key guide](/ed25519-ssh-key/).

## Decoding the Files: What's Inside?

If you open these files in a text editor, you will see very different structures.

### 1. The Public Key (`id_ed25519.pub`)
The public key is stored as a single line of plaintext. You can safely display it using the `cat` command:

```bash
cat ~/.ssh/id_ed25519.pub
```

The output looks like this:
```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIF58hF7p6R1122pX5U69o1u9yH8J30b355y67U7H8b8C your_email@example.com
```

This line is split into three space-separated parts:
1. **Key Type (`ssh-ed25519`):** Identifies the cryptographic algorithm.
2. **Key Data (`AAAAC3...`):** The Base64-encoded representation of your public key (specifically, the 32-byte public key point wrapped in a standard SSH wire format structure).
3. **Comment (`your_email@example.com`):** A custom label for identification.

### 2. The Private Key (`id_ed25519`)
The private key is much longer and is protected by formatting headers. You should **never** share or paste this content. If you open it, it looks like this:

```text
-----BEGIN OPENSSH PRIVATE KEY-----
b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAAAMwAAAAtzc2gtZW
QyNTUxOQAAACBeVIRe6ekdZdZqV+VOvaNcvch/Cd9G9+ecuu1Ox/G/AgAAAIiV57bBlee2
wQAAAAtzc2gtZWQyNTUxOQAAACBeVIRe6ekdZdZqV+VOvaNcvch/Cd9G9+ecuu1Ox/G/Ag
...
-----END OPENSSH PRIVATE KEY-----
```

#### The OpenSSH Private Key Format
In the past, SSH keys were stored in PEM formats defined by OpenSSL. Modern OpenSSH uses its own custom format, indicated by the `-----BEGIN OPENSSH PRIVATE KEY-----` header.

The data block contains:
* **Cipher Name:** Indicates the encryption algorithm (e.g., `aes256-ctr` if a passphrase was set, or `none` if empty).
* **KDF Name:** The Key Derivation Function used to turn your passphrase into an encryption key. OpenSSH uses **bcrypt** (specifically `bcrypt_pbkdf`), which is highly resistant to brute-force attacks.
* **Salt and Rounds:** The random salt and number of iterations used by the KDF to slow down attackers trying to guess your passphrase.
* **Secret Payload:** The encrypted (or plaintext) block containing both your private seed and your public key.

## Strict Permissions: Protecting Your Keys

SSH client programs are highly sensitive about security. If your private key file can be read by other users on your operating system, the `ssh` command will refuse to run, outputting a warning:

```text
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for '/home/user/.ssh/id_ed25519' are too open.
It is required that your private key files are NOT accessible by others.
This private key will be ignored.
```

To fix this, you must configure tight file permissions.

### 1. Setting Permissions on Linux and macOS
On Unix-like systems, use the `chmod` (change mode) command to restrict permissions:

```bash
# Set private key to read/write by owner only (600)
chmod 600 ~/.ssh/id_ed25519

# Set public key to read/write by owner, read-only by others (644)
chmod 644 ~/.ssh/id_ed25519.pub

# Set the hidden .ssh directory to read/write/execute by owner only (700)
chmod 700 ~/.ssh
```

### 2. Setting Permissions on Windows (PowerShell)
Windows uses Access Control Lists (ACLs) rather than Unix-style octal permissions. If you use Windows OpenSSH, you must remove inheritance and restrict read access to your own user account.

Run the following commands in an elevated PowerShell session:

```powershell
# Define the path to the private key
$path = "$env:USERPROFILE\.ssh\id_ed25519"

# Disable inheritance and remove inherited rules
icacls $path /c /t /inheritance:d

# Grant full control to the current user
icacls $path /c /t /grant:r "$($env:USERNAME):(F)"

# Remove access for "Everyone" and "Users"
icacls $path /c /t /remove "Everyone"
icacls $path /c /t /remove "Users"
```

## Key Management with SSH Agent

Typing your key's passphrase every single time you connect to a server or run `git fetch` can quickly become annoying. To solve this, OpenSSH includes an in-memory credential cache called the **SSH Agent** (`ssh-agent`).

### How `ssh-agent` Works
The SSH Agent runs in the background. You add your private key to the agent once, type your passphrase to decrypt it, and the agent holds the decrypted key in your computer's RAM. When you connect to a server, the `ssh` client communicates with the agent to perform the challenge signing, meaning your private key never has to be loaded from disk or decrypted again.

### Using the Agent (Step-by-Step)

#### Step 1: Start the SSH Agent
* **On Linux/macOS:**
  ```bash
  eval "$(ssh-agent -s)"
  ```
* **On Windows (PowerShell):**
  Ensure the SSH Agent service is enabled and running:
  ```powershell
  Set-Service -Name ssh-agent -StartupType Automatic
  Start-Service ssh-agent
  ```

#### Step 2: Add Your Key
Run the `ssh-add` command to register your key:

```bash
ssh-add ~/.ssh/id_ed25519
```

Type your passphrase when prompted. The agent will confirm:
`Identity added: /home/user/.ssh/id_ed25519 (your_email@example.com)`.

### A Warning About Agent Forwarding (`ssh -A`)
OpenSSH allows you to forward your local agent to a remote server using the `-A` flag. This lets you use your local keys on the remote server to log into subsequent machines. 

**Warning:** If you forward your agent to a server where another user has root (admin) access, that administrator can access the Unix domain socket of your forwarded agent and authenticate to other servers as *you*. 

**Mitigation:** Only use agent forwarding to servers you completely trust, or use OpenSSH's newer `ProxyJump` option instead of forwarding.

## Troubleshooting Common SSH Authentication Failures

If you cannot connect using your Ed25519 key, use this checklist to diagnose and resolve the issue.

### 1. Enable Verbose Debugging
The most effective way to troubleshoot is to run the connection in verbose mode by appending `-v`, `-vv`, or `-vvv` to your SSH command:

```bash
ssh -v git@github.com
```

Look for lines starting with `debug1:`. In particular, search for:
* `debug1: Authentications that can continue:` (lists the methods the server accepts, like `publickey`).
* `debug1: Offering public key: /home/user/.ssh/id_ed25519` (shows if the client is finding your key).
* `debug1: Authentications failed` (indicates that the key was offered but rejected).

### 2. Verify the Server's `authorized_keys` Permissions
If you are logging into a server you control, the remote server's permissions must be restricted. SSH will refuse to use `authorized_keys` if the remote user's directory is writable by others.

Connect using a password or console and run these commands on the server:

```bash
chmod 700 ~
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
```

### 3. Check `sshd_config` Configuration
Ensure the remote server's daemon allows public key authentication. Open `/etc/ssh/sshd_config` on the server and verify that the following lines are set:

```text
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
```

If you made changes, restart the SSH service:
`sudo systemctl restart ssh` or `sudo service ssh restart`.

## Frequently Asked Questions (FAQs)

### Q1: What is the difference between id_ed25519 and id_ed25519.pub?
`id_ed25519` is your private key, which must remain secret and never leave your computer. `id_ed25519.pub` is the matching public key, which you install on remote servers (like GitHub or AWS) so they can authenticate your connection.

### Q2: What happens if I forget my SSH key passphrase?
If you forget your passphrase, you cannot decrypt your private key file. SSH passphrases cannot be reset or recovered. You will have to generate a new key pair (`id_ed25519` and `id_ed25519.pub`), add the new public key to your servers, and delete the old one.

### Q3: Why does SSH block my connection with "Permissions are too open"?
SSH requires your private key to be readable only by you. If other user accounts on your computer can read the private key file, the SSH client ignores the key to prevent unauthorized access. You must restrict the file permissions to `0600` (`chmod 600`).

### Q4: Can I use the same id_ed25519 key on multiple computers?
While mathematically possible to copy the files, it is a security risk. If one device is compromised, all your servers are exposed. It is highly recommended to generate a unique key pair for each of your computers (e.g., laptop, desktop, office workstation) and add all of their public keys to your server's `authorized_keys` file.

### Q5: How do I change the passphrase of an existing key?
You can change the passphrase of your `id_ed25519` private key without generating a new key pair by running:
`ssh-keygen -p -f ~/.ssh/id_ed25519`
This prompts you for the old passphrase, then asks for the new passphrase and saves the key.
