*Last updated: June 18, 2026*

If you have ever set up secure server access, configured Git repositories, or investigated how secure messaging works, you have run into key pairs. These pairs consist of a public key and a private key. They form the mathematical foundation of asymmetric cryptography.

> **Featured Snippet: What is the difference between a public key and a private key?**
> A public key is shared openly and is used to encrypt data or verify digital signatures. A private key is kept strictly secret by its owner and is used to decrypt data encrypted with the public key or to generate digital signatures. They form a secure asymmetric cryptographic pair.


Although these keys are mathematically linked, they serve opposite purposes. Understanding the distinction between them is essential to maintaining your security online. A single mistake, such as exposing a private key or confusing the files, can compromise your servers, source code repositories, and secure communication channels.

In this guide, we will compare public and private keys in detail, explain how they are mathematically bound together, outline key distribution methods, and detail critical security rules for protecting your keys.

## The Asymmetric Key Pair Concept

Public key cryptography uses two different keys to secure data. This is different from symmetric cryptography, which uses a single password for both locking and unlocking files.

Think of it like a secure physical lockbox with two separate slots:
* **The Public Key Slot:** Anyone can insert a document into the box through the slot. This represents encryption. You can publish this slot to the world.
* **The Private Key Door:** Only you hold the key to open the door at the back of the box and retrieve the documents. This represents decryption. You must keep this key secret.

Because the capability to write is separated from the capability to read, you can share your public key over insecure networks like the internet. Even if an attacker captures your public key, they cannot use it to read your encrypted messages or sign documents on your behalf.

## Public Key vs. Private Key Comparison

Below is a detailed comparison of public and private keys:

| Property | Public Key | Private Key |
| :--- | :--- | :--- |
| **Access Level** | Open to anyone. Shared publicly. | Kept secret. Stored locally. |
| **Primary Function** | Encrypts data and verifies signatures. | Decrypts data and creates signatures. |
| **Typical File Extension** | Often `.pub` or `.pem` | Often no extension or `.key` / `.ppk` |
| **Network Presence** | Transmitted over the network. | Never sent over the network. |
| **Derivation Path** | Derived from the private key. | Generated from random entropy. |
| **Security Risk** | No security risk if exposed. | Total compromise if exposed. |

## The Mathematical Link: One-Way Derivation

A public key is not generated independently. It is derived directly from the private key through a mathematical process known as a **one-way function**.

### The Math Behind the Link
In elliptic curve cryptography, such as Curve25519 used by Ed25519, the private key starts as a 32-byte random seed of entropy. The system clamps this seed and treats it as a large integer scalar, which we can call $s$. 

The public key is generated by multiplying the private scalar $s$ by a standardized starting point on the curve, known as the **Base Point ($B$)**:
$$\text{Public Key Point } A = s \cdot B$$

In this system, multiplication is simple. However, division is extremely difficult. If you are only given the public key point $A$ and the public base point $B$, calculating the secret scalar $s$ is known as the **Elliptic Curve Discrete Logarithm Problem (ECDLP)**. Solving this problem requires billions of years using modern supercomputers, meaning the public key is secure to share.

For more details on this process, see our guide on [how Ed25519 key generation works](/blog/how-ed25519-key-generation-works/).

## Roles in Secure Communication

Depending on the task, keys play different roles:

### 1. Data Encryption
When you want to send encrypted data, you use the recipient's public key to secure it. Only the recipient's private key can decrypt the data.
* **Encryption:** Done with the Public Key.
* **Decryption:** Done with the Private Key.

### 2. Digital Signatures
When you want to sign a message to prove your identity, you use your private key to generate a signature. Anyone can use your public key to verify that the signature was created by you.
* **Signing:** Done with the Private Key.
* **Verification:** Done with the Public Key.

Learn more about this process in our [digital signature vs encryption guide](/blog/digital-signature-vs-encryption/).

## Key Distribution Models and Best Practices

To connect with other systems, you must distribute your public key while keeping your private key local:

* **SSH Key Setup:**
  You save your private key in the local file `~/.ssh/id_ed25519`. You copy the public key file `~/.ssh/id_ed25519.pub` to the remote server and append it to `~/.ssh/authorized_keys`. The server uses it to verify your login attempts. For detailed details on these files, check out our [id_ed25519 article](/blog/id_ed25519/).
* **GitHub Integration:**
  You upload your public key to your GitHub account settings. This allows you to securely clone and push to your repositories over SSH.
* **TLS Certificates:**
  Websites present a certificate containing their public key. Your browser verifies the certificate and uses the public key to negotiate a secure connection.

## Critical Security Rules for Key Safety

To keep your credentials secure, follow these five essential security rules:

1. **Never Share Your Private Key:**
   Do not email your private key, paste it into chat applications, or upload it to repository hosting sites. If a service provider asks for your private key, it is likely a scam or a poorly designed system.
2. **Use Strong Passphrases:**
   Encrypt your private key file with a secure passphrase when generating it. This adds a layer of defense if your device is compromised.
3. **Restrict File Permissions:**
   Ensure your private key file has tight operating system permissions. On Linux and macOS, use `chmod 600`. On Windows, configure inheritance rules using PowerShell.
4. **Isolate Keys Across Devices:**
   Do not copy a single private key across all your devices. Generate a unique key pair on each computer, and authorize each public key independently on your servers.
5. **Use a Credential Agent:**
   Use tools like the SSH agent or PuTTY's Pageant to load decrypted keys into memory, preventing you from having to type passphrases repeatedly. For more information, read our [putty-ed25519 guide](/blog/putty-ed25519/).

## Code Example: Key Pairs in Python

Below is a Python script using the `cryptography` library to generate an Ed25519 key pair, demonstrating that the public key is extracted directly from the private key:

```python
from cryptography.hazmat.primitives.asymmetric import ed25519
from cryptography.hazmat.primitives import serialization

# 1. Generate the Private Key
private_key = ed25519.Ed25519PrivateKey.generate()

# 2. Extract the Public Key from the Private Key
public_key = private_key.public_key()

# 3. Serialize Private Key to PEM format
private_pem = private_key.private_bytes(
    encoding=serialization.Encoding.PEM,
    format=serialization.PrivateFormat.PKCS8,
    encryption_algorithm=serialization.NoEncryption()
)

# 4. Serialize Public Key to OpenSSH format
public_openssh = public_key.public_bytes(
    encoding=serialization.Encoding.OpenSSH,
    format=serialization.PublicFormat.OpenSSH
)

print("Private Key PEM:")
print(private_pem.decode()[:150] + "...\n")

print("Public Key OpenSSH:")
print(public_openssh.decode())
```

## Frequently Asked Questions (FAQs)

### Q1: Is it possible to generate a private key if you have the public key?
No. Elliptic curve math makes the derivation path a one-way operation. Calculating the private key from the public key is mathematically infeasible due to the complexity of the discrete logarithm problem.

### Q2: What happens if I lose my private key?
If you lose your private key, you lose access to any server or system that relies on it. You cannot recover or reset the key. You must generate a new key pair, install the new public key on your servers, and remove the lost public key.

### Q3: What happens if someone gets my public key?
Nothing. The public key is designed to be shared openly. An attacker cannot use your public key to log into your servers or read your encrypted data. They can only use it to send encrypted files to you or verify your digital signatures.

### Q4: Can I use the same public key for multiple servers?
Yes. You can install your public key on thousands of remote servers. This allows you to log into all of them using your single local private key.

### Q5: How do I verify that a private key matches a public key?
In OpenSSH, you can extract the public key from the private key file to compare them. Run:
`ssh-keygen -y -f ~/.ssh/id_ed25519`
If the output matches the contents of your `id_ed25519.pub` file, the keys are a matching pair.
